# 张文婧 · 项目面试题 · AI Agent 全流程开发工作流体系

> 说明：本题库基于「Rong Workflow Skills」项目文档定制。核心亮点是 31 个 Claude Code Skills 编排的 S0-S14 全流程 iOS App 开发工作流，含双并行区、三级变更管理、Harness 持久化状态和增量开发模式。共 40 题。

---

## 1. 项目动机与价值

### Q1. 这个项目解决什么问题？为什么要做？

**面试官意图**：验证项目不是"为了炫技"而是解决真实痛点。

**参考答案**：

> 我在用 Claude Code 做 Expo + React Native 项目时发现一个反复出现的问题：**AI 能写代码，但不会自己管项目。**
>
> 具体三个痛点：
> 1. **每次新会话都要重新交代上下文**——上次做到 S8 前端第 3 个页面了，新开会话它不知道，得从头讲，浪费 10-15 分钟。
> 2. **阶段之间的上下游依赖靠人脑串联**——PRD 改了一个接口定义，我得手动通知前端和后端两边的 Agent 都更新。
> 3. **变更需求时不知道影响范围**——加一个新字段，到底要改 ERD、类型定义、接口、联调用例中的哪几个？只能我自己查。
>
> 所以我设计了这套工作流：**把"人脑串联"的工作编码成 31 个 Skill 之间的依赖关系，让 Agent 自己知道"上一步给了我什么、我该输出什么、下一步谁来接"。**

**可能追问**：市面上有没有类似的东西？和 Cursor 的 Agent 模式有什么区别？

---

### Q2. 为什么是 31 个 Skill，不是一个超大 Prompt 搞定？

**面试官意图**：考察系统拆分的理解。

**参考答案**：

> 我一开始试过把整个流程写成一个 8000 字的超长 Prompt，遇到三个问题：
>
> 1. **注意力稀释**——Prompt 太长，Agent 对中间阶段的指令执行质量明显下降，经常跳步骤。
> 2. **无法并行**——一个 Prompt 串行执行，没法让设计和架构同时跑。
> 3. **无法复用**——只想做后端架构时，得手动删一大堆无关指令。
>
> 拆成 31 个独立 Skill 后：每个 Skill 只关注一个阶段，Prompt 短且精准；并行区的 Skill 可以在不同会话里同时运行；增量模式直接调 S8/S9 跳过不需要的阶段。每个 Skill 有明确的输入/输出契约，依赖关系是显式声明的。

**可能追问**：31 个 Skill 之间的依赖管理成本不高吗？

---

### Q3. 你怎么定义这个项目的成功？有量化指标吗？

**面试官意图**：看有没有产品思维。

**参考答案**：

> 三个维度：
>
> 1. **效率提升**：从想法到可提审版本的周期。之前调研+PRD+设计+架构阶段需要 5-7 个工作日；用工作流后压缩到 1-2 天——因为并行区 1 把设计和架构同时推进，Agent 自动做文档生成。
>
> 2. **上下文损耗**：跨会话需要重新交代上下文的次数。之前每次新会话花 10-15 分钟重新描述项目状态；现在 Agent 读 Harness 三文件就能恢复上下文，基本零损耗。
>
> 3. **变更成本**：需求变更时的返工量。之前改一个接口定义，手动检查 4-5 个文件是否需要同步更新；现在 `/change-request` 自动评估影响范围，精确告诉我哪些阶段要回退。
>
> 这些是体感对比，不是严格 A/B 实验——但效率差距是量级上的，从"天级"到"小时级"。

**可能追问**：这些效率提升数据有没有记录？

## 2. 架构与阶段设计

### Q4. 14 个阶段是怎么拆的？有什么原则？

**面试官意图**：考察系统设计和抽象能力。

**参考答案**：

> 三个原则：
>
> **原则 1：每个阶段必须有明确的输入/输出契约。** S2 输出 PRD，S5 输出 Design Spec，S6 输出后端 PRD——下游 Skill 严格依赖上游的输出格式。如果一个阶段的输出不明确，下游就没法接。
>
> **原则 2：每个 Review Gate 对应一次不可自动化的人类判断。** PRD 写完要人确认（S2 High Risk）、设计稿要人审（S5）、接口定义要人过（S6）——这些判断质量直接影响后续所有工作，不能让 AI 自己决定"过了"。
>
> **原则 3：并行/串行的边界对齐真实开发流程。** 设计和架构可以并行（互不依赖）→ 并行区 1。前端和后端可以并行（通过接口契约解耦）→ 并行区 2。但联调必须等两边完成 → S10 串行。这不是我发明的，是真实开发的约束。

**可能追问**：有没有阶段是后来加的？

### Q5. S2 PRD 为什么标记为 High Risk？

**面试官意图**：考察风险控制意识。

**参考答案**：

> PRD 是所有下游工作的地基。如果 PRD 有一个错误的需求定义没被发现，到 S8/S9 开发阶段才暴露，返工成本是 PRD 修改成本的 **10 倍以上**。
>
> 其他阶段也有风险，但影响范围是收敛的：S5 有问题只影响 S8 前端，S6 有问题只影响 S9 后端。而 S2 PRD 的问题会**扩散到下游所有阶段**——设计、架构、前端、后端、联调、测试全部受影响。
>
> 所以 S2 的规则是：**AI 写完 PRD 后必须停下来等人确认，不能自动流入下一步。** 这是整个工作流里唯一一个"强制暂停"节点。

**可能追问**：人怎么审 PRD？有审核清单吗？

## 3. 并行设计

### Q7. 两个并行区有什么区别？

**面试官意图**：**项目最有设计感的部分。**

**参考答案**：

> 本质区别是**隔离粒度**：
>
> **并行区 1（S4a 设计 + S4b 架构）= 文档级并行**。两边都在写文档（Design Spec 和后端 PRD），不涉及代码。不需要代码隔离，只需保证两份文档在 S7 整合时不冲突。两个 Agent 分别在不同会话里工作，各自输出文档，S7 负责合并和冲突检测。
>
> **并行区 2（S8 前端 + S9 后端）= 代码级并行**。两边都在写代码，在同一个仓库里。如果前端 Agent 改了 `package.json`，后端 Agent 也改了 `package.json`，就会冲突。隔离方式：**Git Worktree**——每个 Agent 在独立的工作目录里开发，共享 .git 历史但文件完全隔离。

**可能追问**：并行区 2 的两个 Agent 怎么共享接口定义？

---

### Q8. 为什么选 Git Worktree 做并行隔离？

**面试官意图**：考察技术选型的思考过程。

**参考答案**：

> 对比了三种方案：
>
> | 方案 | 问题 |
> |------|------|
> | 分两个仓库 | 类型共享断裂，联调成本高 |
> | 用分支来回切 | Claude Code 会话内 git checkout 有状态污染风险 |
> | **Git Worktree** | **同一个仓库的两份独立工作目录，文件隔离但 .git 共享** |
>
> Worktree 的优势：前端在 `worktrees/frontend/` 写 Expo 页面，后端在 `worktrees/backend/` 写 Node 接口，类型定义放在主仓库共享，S10 联调时两个 worktree 的代码自然合并回 main。

**可能追问**：Git Worktree 踩过什么坑？

---

### Q9. Worktree 踩过的坑？

**面试官意图**：验证你真的在用。

**参考答案**：

> 三个坑：
>
> 1. **.gitignore 不同步**——前端需要忽略 `.expo/`，后端需要忽略 `dist/`。一开始没注意，后端的 `.expo/` 被提交了。后来在 `using-git-worktrees` Skill 里加了 `.gitignore` 验证步骤。
>
> 2. **创建后的初始状态不干净**——主仓库有未提交改动时，worktree 会继承这些"脏"文件。后来加了"创建前自动 stash、创建后验证 baseline"的流程。
>
> 3. **合并回主仓库时的冲突**——S8 和 S9 都可能改 `README.md` 或 `package.json`，合并必然冲突。目前 S10 联调阶段手动 resolve。未来计划让 Agent 自动识别常见 merge conflict 模式。

**可能追问**：类型定义怎么在两个 worktree 间共享？

## 4. 变更管理

### Q10. 三级变更管理是怎么工作的？举个具体例子。

**面试官意图**：**核心差异化，验证设计深度。**

**参考答案**：

> 假设我在 S9 后端开发到一半时，产品说"用户资料里要加一个'生日'字段"。
>
> 我输入 `/change-request`，Agent 做三件事：
>
> **Step 1 · 变更分类**：扫描受影响阶段——S6 ERD 要加一列、S8 前端要加 DatePicker、S9 接口要接收新字段、S10 联调要验证一致性。影响 4 个阶段 → 判定 **L2 跨阶段变更**。
>
> **Step 2 · 生成回退路径**：
> - S6：ERD 加 `birthday DATE NULL` + migration script
> - S9：DTO 加 `birthday` + 数据库 migration
> - S8：用户资料页加 DatePicker + 类型定义更新
> - S10：新增联调验证用例
>
> **Step 3 · 执行策略**：建议顺序 S6 → S9 → S8 → S10，同时在 decisions.md 记录变更原因和影响范围。
>
> 如果不是"加字段"而是"把 PostgreSQL 换成 MongoDB"→ **L3 架构级变更**，Agent 建议回退到 S4b 重做架构方案。

**可能追问**：L1/L2/L3 的边界是 Agent 自动判的还是人定的？

### Q11. 回退路径矩阵怎么设计的？

**面试官意图**：追问变更管理细节。

**参考答案**：

> 本质是一张"每个阶段变更后需要通知哪些下游"的依赖表：
>
> | 变更发生在 | 需要回退/更新的下游 |
> |-----------|-----------------|
> | S2 PRD | S4a, S4b, S5, S6, S7, S8, S9（几乎全部）|
> | S5 Design Spec | S7, S8 |
> | S6 后端 PRD | S7, S9 |
> | S8 前端代码 | S10 |
> | S9 后端代码 | S10 |
>
> 回退不了的情况：**S14 已提交 App Store 审核了要改需求，只能等审核完再发新版**。所以 S12-S14 的变更策略不是回退，而是"记录到下个版本 backlog"。

**可能追问**：v1 没有变更管理的时候是什么情况？

### Q12. v2 为什么要加变更管理？v1 遇到了什么问题？

**面试官意图**：验证迭代过程的真实性。

**参考答案**：

> v1 的时候没有 change-request Skill。结果第一次在 S9 后端开发到一半需求变了——要加一个新接口。我不知道影响范围，只能自己查：ERD 要改吗？前端有没有用到这个接口？联调用例要更新吗？查了一个小时才理清楚。
>
> 这个教训直接催生了 v2 的三级变更管理。**变更管理不是"事后弥补"，是"系统设计的一部分"**——应该在设计之初就考虑"如果需求变了怎么办"，而不是等出了问题再打补丁。

**可能追问**：v2 还加了什么其他改动？

## 5. Harness 状态管理

### Q13. Harness 三文件解决什么问题？

**面试官意图**：考察对 Agent 跨会话问题的理解。

**参考答案**：

> AI Agent 最大的短板是**跨会话记忆丢失**。一个 App 的开发跨越几十次会话，如果每次都要重新理解"技术栈是什么""做到哪了""之前选了什么方案"，效率急剧下降。
>
> Harness 三文件就是 Agent 的**持久化工作记忆**：
>
> | 文件 | 角色 | 解决的问题 |
> |------|------|-----------|
> | AGENTS.md | 项目约束和规则 | "这个项目用 Expo 不用 Flutter"——Agent 不会跑偏 |
> | progress.md | 阶段进度和阻塞点 | "S8 已完成 3/5 页面"——Agent 知道从哪继续 |
> | decisions.md | 关键决策和变更日志 | "选了 PostgreSQL 不选 MongoDB，原因是..."——Agent 不会重复纠结 |
>
> 最关键的设计原则：**Skill 不应该假设上下文，而应该从 Harness 文件和上游输出中读取上下文。** 这样任何 Skill 都可以在全新会话里启动。

**可能追问**：Agent 会不会忘了更新 Harness 文件？

---

### Q14. Agent 忘了更新 Harness 文件怎么办？

**面试官意图**：追问持久化的可靠性。

**参考答案**：

> **会。这是 AI Agent 系统最常见的问题。**
>
> 应对方式：
> 1. **harness-maintainer 是独立 Skill**——不嵌入其他 Skill。其他 Skill 完成后触发它来更新三文件。"更新状态"不依赖"执行任务"的 Agent 记不记得。
> 2. **每个 Skill 结束时有"汇报"步骤**——把输出路径、完成状态、关键决策写成标准格式，harness-maintainer 读这个格式来更新。
> 3. **progress.md 有"当前阶段"和"最后更新时间"**——如果 Agent 发现自己要做的阶段和记录不一致，主动提醒人确认。
>
> 但诚实说，这套机制不是 100% 可靠。Agent 有时还是跳过汇报步骤。当前兜底是人定期检查。Future work 是做自动校验脚本。

**可能追问**：自动校验脚本大概怎么设计？

## 6. 增量模式与复用

### Q15. 增量模式解决什么问题？

**面试官意图**：考察对实际使用场景的理解。

**参考答案**：

> 完整流程（S0-S14）适合从零开始的新项目，但大量场景是"已有一个 App，要加一个功能"。如果每次都从 S1 想法验证开始跑，没人会用——太重了。
>
> 所以 S8/S9 各设计了一个**增量入口**：
> - 跳过 S1-S7 的完整流程
> - 直接读取已有项目的 AGENTS.md 和代码结构
> - 只执行"需求理解 → 设计方案 → 编码 → 约束检查"的轻量子流程
> - 同时在 change-request 里记录变更，保持 Harness 文件一致性
>
> 这是"完整流程太重、小需求跑不动"这个真实痛点逼出来的设计。

**可能追问**：增量模式和完整模式的代码是共享的还是两套？

---

### Q16. 这套工作流的复用价值在哪？

**面试官意图**：考察长期思考。

**参考答案**：

> 三层复用：
>
> 1. **项目级复用**：做下一个 Expo App 时，`/app-launcher` 一键启动，31 个 Skill 自动编排。不需要重新搭工作流。
>
> 2. **方法论复用**：14 个阶段、Review Gate、变更管理、Harness 体系——这些设计模式不依赖具体技术栈。如果团队从 Expo 换到 Flutter，只需改 S8 的 Skill 内容，工作流架构不变。
>
> 3. **AI Agent 系统设计复用**：31 个 Agent 之间的依赖编排、状态持久化、异常处理——这是一套 Agent Pipeline 设计范式，可以迁移到任何需要"多 Agent 协作完成多阶段任务"的场景。

**可能追问**：如果要把这套工作流移植到 Cursor，需要改什么？

## 7. AI Agent 工程

### Q17. Skill 的 SKILL.md 怎么写？有模板吗？

**面试官意图**：考察 Prompt Engineering 的系统性。

**参考答案**：

> 每个 SKILL.md 遵循统一结构：
>
> 1. **Frontmatter 元数据**：名称、描述、触发条件、依赖的其他 Skill
> 2. **角色定义**：这个 Skill 扮演什么角色（PM/设计师/架构师/开发者...）
> 3. **输入契约**：期望的输入格式和来源（如"读取 S2 阶段输出的 PRD"）
> 4. **执行步骤**：有序步骤列表，每步有明确动作和预期输出
> 5. **输出契约**：输出格式和存放位置
> 6. **Review Gate 定义**：什么条件算完成、需要人审核什么
> 7. **异常处理**：输入缺失/格式不对/上游跳过时怎么降级
>
> 最关键：**Skill 不假设上下文，从 Harness 文件和上游输出中读取上下文。**

**可能追问**：有没有 Skill 之间"接口不兼容"的问题？

---

### Q18. 为什么不用 LangChain / LangGraph？

**面试官意图**：考察技术选型的思考深度。

**参考答案**：

> 三个原因：
>
> 1. **目标不同**：LangChain/LangGraph 是"让 LLM 调用工具"，我这套是"让 LLM 按阶段执行软件开发流程"。前者粒度是"一次函数调用"，后者粒度是"写一份 PRD"或"开发一个页面"。抽象层级不匹配。
>
> 2. **原生解决方案**：我的工作流跑在 Claude Code 里，Skills 是它的原生扩展机制。用 LangChain 意味着在外面再套一层框架，增加复杂度但不增加能力。
>
> 3. **状态管理方式不同**：LangGraph 的状态是内存态，会话结束就丢。我用文件系统（Harness 三文件）做持久化，天然跨会话。

**可能追问**：如果要移植到 Cursor，需要改什么？

## 8. 和其他项目的关系

### Q19. 这个工作流体系和你前面四个 Refantasia 项目是什么关系？

**面试官意图**：考察全局视野。

**参考答案**：

> 两个不同层面的工作：
>
> - **Refantasia 四个项目**是"AI 产品"——面向游戏运营的具体业务场景（流失预警、内容生成、数据分析、排期优化），解决的是"游戏运营怎么用 AI"。
>
> - **Rong Workflow Skills**是"AI 工具/基础设施"——面向开发者（包括我自己），解决的是"怎么用 AI Agent 高效做出一个 App"。
>
> 两者的共性是同一个设计哲学：**"AI 做初稿，人做审核"**。Refantasia 项目里运营审核 AI 生成的配置，Workflow 里人审核 AI 写的 PRD 和代码。**我不追求 AI 全自动，我追求 AI 在人的监督下做 80% 的重复工作。**

**可能追问**：做 Refantasia 项目的时候有没有用这套工作流？

### Q20. 这个项目体现了你什么能力？和 AI PM 岗位有什么关系？

**面试官意图**：考察候选人对自身定位的理解。

**参考答案**：

> 三个能力：
>
> 1. **系统设计能力**：不是写了一个 Prompt 就叫"AI 应用"，而是设计了 31 个 Agent 之间的依赖关系、数据流转、状态管理和异常处理。这和做 AI 产品需要的"端到端流水线设计"是同一种能力。
>
> 2. **流程抽象能力**：把"从 0 到 1 做 App"这个模糊过程拆成 14 个阶段、定义输入输出契约、设计 Review Gate——这就是 PM 做需求拆解和优先级排序的核心功。
>
> 3. **AI Agent 深度理解**：Agent 的核心问题不是 Prompt 写得好不好，而是状态管理、并行编排、变更处理。做过 31 个 Skill 的编排之后，我对"Agent 能做什么、不能做什么"有第一手体感——这比"读了几篇 Agent 论文"可靠得多。

## 9. 反思与迭代

### Q21. 这个项目你最不满意的地方是什么？

**面试官意图**：考察批判性思维。

**参考答案**：

> 四个不足：
>
> 1. **没有端到端自动化测试**——验证"整条工作流能跑通"的方式是我自己走一遍，没有 CI 级别的自动化。
>
> 2. **Harness 文件一致性不够可靠**——Agent 有时忘更新 progress.md，靠人定期检查兜底。
>
> 3. **并行区 2 的类型共享不优雅**——前后端在不同 Worktree，共享类型定义靠人手动同步。理想方案是 monorepo + shared types package。
>
> 4. **Prompt 可维护性差**——31 个 SKILL.md 加起来几万字，全局修改一个约定要手动改 31 个文件。应该抽出 base template 让每个 Skill 继承扩展。

### Q22. 你从这个项目学到了什么？

**面试官意图**：收尾题。

**参考答案**：

> 三个认知升级：
>
> 1. **AI Agent 的核心不是 Prompt，是状态管理。** Prompt 解决"单次任务"质量，但真实项目有几十次任务串联，状态（做到哪了、决策了什么）才是关键。Harness 三文件就是被这个认知逼出来的。
>
> 2. **并行 ≠ 同时开始，而是"依赖可解耦"。** 设计和架构能并行，是因为它们的输入都是 PRD、互不需要对方输出。有依赖就不能并行，强行并行只会制造合并冲突。
>
> 3. **变更管理是系统设计的一部分，不是事后弥补。** v1 没有 change-request，第一次遇到需求变更就手忙脚乱。"如果需求变了怎么办"应该在设计之初就考虑。
