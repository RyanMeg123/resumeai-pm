## 1. AI产品体验与行业趋势

### Q1: 你最近觉得最好玩的 AI 产品是什么？（国内外都可以）

**参考答案**：

我说几个不同方向的，都是我真正用过的：
>
**Perplexity Comet 浏览器**（2026.3 上线）—— 这不只是"搜索引擎加 AI"，是把 AI 做进了浏览器本身。你浏览任何网页时可以直接问 AI，Deep Research 能直接生成 PPT 和数据报表，而且它记住你的偏好帮你购物订票。这让我思考：**AI 的最终形态可能不是独立 App，而是一个环境层。**
>
**Cursor / Claude Code** —— 我 4 个项目全部代码都是 Cursor 辅助写的。Agent 模式（描述需求 → 自主改多文件）改变了我的编程习惯：先写注释描述意图，再让 AI 填实现。Claude Code 更适合大规模推理和 GitHub 工作流。两个产品让我确信：**AI 最好的形态不是对话框，而是嵌入工作流。**
>
**Lovable**（AI 全栈应用生成器）—— 一句话就能生成完整 React + Supabase 应用，代码同步 GitHub，支持登录/支付/数据库。2026.3 扩展到不只是建应用，还能做数据分析和 BI。这让"vibe coding"正在成为真正的生产力。
>
**ElevenLabs** —— 语音克隆 + 多语言实时语音 Agent。这对我的项目2图像通道有启发——如果语音也能做到这个水平，运营内容的产能瓶颈就不再是"人"了。
>
国内的话我关注 **Kimi（月之暗面）** 的长 context 能力和 **Coze（字节）** 的 Bot 编排平台。

### Q2: 你怎么看 AI Agent 目前的落地情况？离真正好用还差多远？

**参考答案**：

离"好用"还差两个关键能力：**可靠的 tool calling**和**长期记忆**。
>
当前 Agent 最大的问题是"不稳定"——同一个任务跑 10 次可能有 3 次走错路径。在 demo 里看着很酷，但生产环境需要 99%+ 的成功率。这也是为什么我在四个项目里都用"固定 Workflow"而不是"自主 Agent"——MVP 阶段可预测性比灵活性重要。
>
我认为真正的 Agent 落地路径是"渐进式自主"：先做确定性 Workflow（项目2/4 的当前状态），然后在低风险环节引入 Agent 自主决策（比如让 Agent 自己决定要不要重试），最后才是全自主。不会是一步到位。

### Q3: RAG 和长 context window，你觉得哪个会胜出？

**参考答案**：

**不是替代关系，而是互补**。长 context 解决"能不能塞下"，RAG 解决"该不该塞进去"。
>
即使 context window 到了 10M tokens，你把 12608 条配置全塞进去，LLM 的注意力也会涣散——它不知道哪 5 条和你当前问题最相关。RAG 的价值是"信息压缩"——从 12608 条里精准找出 5 条最相关的。
>
但 RAG 的维护成本高（embedding、索引更新、切片策略），如果场景简单（文档不超过 200 页），直接塞 context 更省事。我的判断是：**大规模知识库用 RAG，小规模文档用长 context，两者混合使用是最佳实践。**

### Q4: 你怎么看 AI 产品的"幻觉"问题？能根治吗？

**参考答案**：

**根治不了，但可以工程化地管理。** 我的四个项目就是四种管理幻觉的方法：
>
1. 项目1：LLM 归因要求携带 evidence 字段引用原始数据，校验不过就降级。
2. 项目2：术语用词典强制替换，不靠 LLM"听话"。
3. 项目3：LLM 只输出 Intent JSON，SQL 由编译器生成——编译器不会幻觉。
4. 项目4：CSP 求解器产出排期，LLM 只做解释——解释可以错，排期不会错。
>
共同模式是：**把 LLM 的输出空间收窄到一个可验证的范围**。输出空间越小，幻觉越少。

### Q5: 你怎么看 AI 对产品经理这个职业的影响？PM 会被取代吗？

**参考答案**：

PM 不会被取代，但**不会用 AI 的 PM 会被取代**。
>
AI 能替代的是 PM 工作中的"执行层"——写 PRD 模板、做竞品表格、整理用户反馈。但 PM 的核心价值——需求判断（该不该做）、优先级排序（先做什么）、跨团队推动（说服谁）——这些需要对业务的深度理解和人际判断力，AI 暂时做不了。
>
我对自己的定位是：**用 AI 武装自己的 PM 能力**。比如我用 Cursor 写 PoC、用 LLM 做竞品分析初稿、用 ComfyUI 做原型图——AI 让我一个人能做原来需要 3 个人做的事。

### Q6: 如果给你无限预算做一个 AI 产品，你最想做什么？

**参考答案**：

**面向中小游戏公司的"AI 运营中台"**。把我这四个项目变成一个 SaaS：接入任意游戏的数据库 → 自动建语义层 → 流失预警 + 内容生成 + 自助 BI + 排期优化一站式解决。
>
为什么是中小游戏公司？因为大厂有自己的数据团队，中小公司运营团队 3-5 人但数据需求和大厂一样复杂。一个"AI 运营中台"能让 3 个人做到 30 人的效果。
>
最大的技术挑战是"语义层的自动化构建"——不同游戏的表结构完全不同，怎么从原始 schema 自动推导出业务语义层。这需要 LLM + schema 分析 + 行业知识库的结合。

### Q7: 你怎么评估一个 AI 功能值不值得做？

**参考答案**：

三个标准：**频率 × 痛度 × AI 增量**。
>
- **频率**：这个需求多久出现一次？日均 5 次以上才值得做产品化。
- **痛度**：现有方案有多痛？"手工拼 JSON 2 小时"是高痛度，"多点两次鼠标"是低痛度。
- **AI 增量**：AI 方案比规则方案好多少？如果规则就能解决 90%，剩下 10% 用 AI 的 ROI 可能不够。
>
我在做项目优先级时就用了这个框架：流失预警（日均影响 + 高痛 + AI 归因有增量）> 运营 Copilot（日均 + 高痛 + RAG 有增量）> 对话 BI（日均 + 中痛 + 语义层有增量）> 活动排期（月均 + 中痛 + CSP 有增量）。

### Q8: 你怎么看 Prompt Engineering 这个职业？会消失吗？

**参考答案**：

**作为独立职业会消失，作为 PM/工程师的必备技能会长期存在。** 就像"会写 SQL"不是一个独立职业但每个数据分析师都要会。
>
Prompt 工程正在被产品化——Cursor 的 Tab 补全你不需要写 Prompt，Claude 的 Artifacts 你也不需要写 Prompt。但在 B 端 AI 产品里，设计 System Prompt、Few-shot 策略、输出 Schema 仍然是 PM 的核心工作。我四个项目里每一个都有精心设计的 Prompt 架构。

### Q9: 你用过哪些 AI 编程工具？体验怎么样？

**参考答案**：

主力用 **Cursor**，也在用 **Claude Code**（Anthropic 的终端 AI 编程工具），试过 GitHub Copilot、Windsurf、Cline。
>
Cursor 的 Agent 模式（描述需求 → 自动改多个文件）是编辑器内体验最好的。Claude Code 更适合大规模项目——它直接在终端里运行，能做 git 操作、跑测试、推 PR，适合复杂的重构任务。另外最近 **Lovable** 类 vibe coding 工具也值得关注——一句话生成全栈应用的效率非常惊人。
>
我 4 个项目里大约 60% 的代码是 AI 辅助写的，但**架构决策和 Prompt 设计全是我自己做的**——AI 写不好这些。最大的感受是：AI 编程工具让"实现"变便宜了，但让"设计"变得更重要了。

### Q10: 你怎么看"AI 原生产品"和"AI 功能嵌入"的区别？

**参考答案**：

**AI 原生产品**从第一天就围绕 AI 能力设计用户体验（如 ChatGPT、Midjourney、Cursor）；**AI 功能嵌入**是在现有产品里加一个 AI 按钮（如 Notion AI、飞书智能助手）。
>
我的四个项目里，对话式 BI 是 AI 原生（没有 LLM 就没有产品），其他三个更偏"AI 功能嵌入"——流失预警本质是 CRM 工具加了 AI 归因，Copilot 本质是运营后台加了 AI 生成。
>
我的观察是：**B 端产品更适合"嵌入"，C 端产品更适合"原生"。** 因为 B 端用户已有工作流（运营已经在用 Game Panel），AI 要融入而不是颠覆；C 端用户没有固定工作流，AI 可以定义全新的交互范式。

### Q11: 你怎么看开源模型 vs 闭源 API？什么时候选哪个？

**参考答案**：

选型四象限（2026 最新模型）：
>
| | 低成本敏感 | 高成本敏感 |
|---|---|---|
| **低定制需求** | 闭源 API（GPT-5.4 / Claude Sonnet 4.6） | 开源小模型（Qwen3.5-7B / GLM-5） |
| **高定制需求** | 闭源 API + 长 Prompt | 开源 + 微调（DeepSeek-V3.2 / Qwen3.5） |
>
2026 年最大的变化是 **DeepSeek-V3.2 把开源模型的性价比拉到了极致**——$0.28/百万 token，接近 GPT-5.4 的 1/18。我的项目里 LLM 用的是 aihubmix（闭源 API），因为 MVP 阶段优先验证方向。但 V2 我计划评测 DeepSeek-V3.2 和 Qwen3.5 做成本优化——特别是 Stage 1 意图解析这种简单任务，用小模型就够了。
>
LoRA 训练我用的是开源栈（SDXL + kohya_ss），因为画风定制必须微调，闭源 API 做不到。

### Q12: 你怎么看 AI 产品的定价策略？

**参考答案**：

AI 产品定价的核心矛盾是"边际成本不为零"——每次调用都有 LLM 成本。传统 SaaS 的边际成本趋近于零。
>
三种策略：
1. **按调用量计费**（如 OpenAI API）：适合开发者，不适合最终用户（用户怕用多了被扣钱）。
2. **包月 + 用量上限**（如 ChatGPT Plus $20/月）：用户心智简单，但要控制滥用。
3. **按价值计费**（如 Cursor Pro "帮你省了多少工程师工时"）：最理想但最难量化。
>
我的项目如果产品化，我会选**包月 + 模块拆分**：基础 BI 模块免费（拉客），Copilot 模块付费，ComfyUI 图像生成按调用量额外计费（因为 GPU 成本高）。

### Q13: 你对 AI 安全和对齐（alignment）怎么看？在产品里怎么考虑？

**参考答案**：

在 B 端 AI 产品里，安全不是"AI 是否说了有害内容"（C 端问题），而是**"AI 是否做了越权的事"**。
>
我的实践：
- 项目3 的 9 层 SQL 安全沙盒——防止 AI 生成的 SQL 做写操作或扫全表。
- 项目2 的术语强制替换——防止 AI 自由翻译造成术语漂移。
- 项目4 的硬约束 100% 满足——AI 排期的版本兼容性由求解器保证，不靠 LLM。
>
核心原则：**AI 可以建议，但不能自主执行高危操作**。所有写入操作都需要人工确认。

### Q14: 你觉得 AI PM 和传统 PM 最大的区别是什么？

**参考答案**：

三个区别：
>
1. **确定性 vs 概率性**：传统产品"按钮点了一定生效"，AI 产品"80% 的时候对，20% 的时候错"。PM 要设计"错了怎么兜底"。
2. **需求文档不同**：传统 PRD 写"输入 A 输出 B"，AI PRD 要写"输入 A 有 85% 概率输出 B、10% 概率输出 C、5% 概率失败，失败时降级为 D"。
3. **评测驱动**：传统产品上线看埋点数据就行，AI 产品需要回归测试集、人工盲评、影子模式、A/B 测试——评测工程量和开发工程量一样大。

---

### Q15: 你怎么看 AI 产品的"用户教育"问题？用户不知道 AI 能做什么怎么办？

**参考答案**：

两个策略：**预置示例 + 渐进引导**。
>
项目3 对话 BI 里我在侧栏预置了 8 条 sample queries——用户第一次点就能看到"原来可以这样问"。项目2 运营 Copilot 里我做了 10 个"灵感 chip"——"春节汉服 / 万圣巫女 / 海边度假"，降低写 prompt 的认知门槛。
>
核心是**不要让用户面对空白输入框**。空白输入框是 AI 产品最大的用户流失点——用户不知道该输什么，就不用了。

### Q16: 如果模型调用成本突然涨 10 倍，你会怎么应对？

**参考答案**：

三步应对：
1. **路由分级**：把简单任务（意图解析、格式校验）路由到小模型（7B），只有复杂任务（归因、创作）走大模型。
2. **缓存**：相同/相似的查询走缓存不调 LLM。对话 BI 里很多查询是重复的。
3. **模板降级**：LLM 日预算耗尽后，自动切到规则模板——项目2 的"术语替换 + 模板填充"不需要 LLM 也能出 70% 质量的结果。

### Q17: 你怎么看现在的 AI 创业机会？

**参考答案**：

**应用层还有巨大机会，但要选"AI 真的有增量"的场景**。
>
我看好三个方向：
1. **垂直行业 BI**（我的项目3 方向）：通用 BI 工具不懂行业术语，垂直 BI + 语义层有壁垒。
2. **运营自动化**（项目2 方向）：B 端运营的重复劳动量巨大，AI 替代 ROI 清晰。
3. **设计辅助**（项目2 图像通道方向）：策划/设计师需要快速出方案，ComfyUI + LoRA 的定制化能力是刚需。
>
不看好的：纯 chatbot（差异化太难）、AI 写小说/写诗（变现难）、做通用 Agent（成熟度不够）。

### Q18: 你最近在学什么 AI 相关的新技术？

**参考答案**：

四个方向：
1. **MCP（Model Context Protocol）**：Anthropic 提出的让 AI 连接外部工具的标准协议，现在已经被 Cursor、Claude Code、Perplexity 等主流产品采用。我在想能不能用 MCP 把项目3 对话 BI 做成一个标准的 data source connector。
2. **Claude Code + Agent SDK**：Anthropic 的终端编程工具和 Agent 开发框架。Agent SDK 让你能构建自定义的 AI Agent，比我之前用的固定 Workflow 灵活很多。
3. **SDXL Turbo / Lightning**：4 步出图，能把项目2 的 ComfyUI 推理从 3-8 分钟压到几十秒。
4. **DeepSeek-V3.2 / V4 评测**：V3.2 开源模型的性价比已经是碾压级别的（$0.28/百万 token），V4 即将发布。这对我的项目做成本优化有直接价值。

### Q19: 你怎么处理 AI 项目中"数据不够"的问题？

**参考答案**：

四种策略，按成本从低到高：
1. **先用规则跑通**：项目1 MVP 用规则引擎而不是模型，就是因为标签数据不够。
2. **用 LLM 做数据增强**：项目3 的回归测试集里"用 LLM 反推运营意图"就是一种数据增强。
3. **迁移学习**：项目2 的 LoRA 训练只用了 130 张，但基础模型（SDXL）已经在几十亿张图上预训练过了。
4. **用户反馈闭环**：项目3 的 👍/👎 反馈直接变成 few-shot 样例和回归测试集——产品上线后数据会越来越多。
