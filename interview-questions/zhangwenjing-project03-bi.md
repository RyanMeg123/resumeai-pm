# 张文婧 · 项目面试题 · AI 游戏数据对话式 BI

> 说明：本题库基于「AI 游戏数据对话式 BI」项目文档定制。核心亮点是"两阶段架构（LLM 输出 Intent JSON → 确定性编译器生成 SQL）"+ 5 实体语义层 + 9 层 SQL 安全沙盒。共 50 题。

## 1. 项目背景与动机

### Q1. 为什么做对话式 BI？核心痛点是什么？

**面试官意图**：验证需求的真实性。

**参考答案**：

> 核心痛点是"非技术同学不会写 SQL，但临时数据需求天天有"。具体有五个表现：
>
> 1. 运营/策划日均 5-10 次临时查询，全部排研发，平均等 1-2 天，决策窗口经常错过。
> 2. 研发每周被 15-25 条临时数据工单打断，严重拖累产品迭代。
> 3. `players` 表 64 列里 40+ 是 JSON 宽字段，研发自己写 `JSON_EXTRACT` 路径都常写错。
> 4. 业务口径不统一——"流失"在运营、策划、管理层三个团队有三种定义，月度报表数字对不上，每月约 1 次事故。
> 5. 客服查单个玩家要登 3 个系统。
>
> 一个典型场景：运营问"100 服里连续 3 天未登录、VIP ≥ 5、且工会里有活跃好友的玩家有多少？"研发说"这得嵌套 JSON_EXTRACT + JOIN，我明天给你"。第二天数据到了，决策时机已过。

**可能追问**：这 15-25 条/周的数据你怎么统计的？

### Q2. 和项目1（流失预警）、项目2（运营Copilot）是什么关系？

**面试官意图**：考察项目矩阵的全局思考。

**参考答案**：

> 三个项目共享四层基础设施：
>
> | 共享资产 | 对话 BI | 项目1 | 项目2 |
> |---------|--------|-------|-------|
> | 业务口径词典 | ✅ 主维护 | ✅ 消费 | ✅ 消费 |
> | 语义层 | ✅ 主维护 | ✅ 消费（特征工程） | — |
> | LLM 网关 | ✅ 共用 | ✅ 共用 | ✅ 共用 |
> | 审计日志模型 | `bi_query_log` | `llm_attribution_log` | `copilot_audit_log` |
>
> 对话 BI 是"数据基础设施"，项目1 和 2 是"业务应用"。比如项目1 定义了"流失 = 14天未活跃 + 付费 ≥ P1"，这个口径写进了对话 BI 的业务口径词典，运营在对话 BI 里问"流失玩家"时自动用同一个定义。反过来，对话 BI 也能查项目1 的效果数据（"上周挽回邮件的打开率怎么样"）。

**可能追问**：如果三个项目的口径冲突怎么办？

### Q3. 为什么不直接用 Superset / Metabase？

**面试官意图**：考察和传统 BI 的差异化定位。

**参考答案**：

> 两个根本差异：
>
> 1. **传统 BI 假设用户知道要看什么**——你得选表、选字段、写过滤条件。但真实场景是用户"连要看什么都不知道"——管理层说"最近 100 服是不是有点不对劲"，这种模糊问题传统 BI 无能为力。对话 BI 的 Agent 会带着用户往下挖。
>
> 2. **JSON 宽表诅咒**：`players` 表 40+ JSON 字段，在 Superset 里根本没法点选——你得自己写 `JSON_EXTRACT(vip, '$.level')`。我的语义层把这些 JSON 路径封装成"虚拟列"（比如 `vip_level`），用户不需要知道底层是 JSON。
>
> 但我不是要替代 Superset——PRD 非目标里明确写了"不替代专业 BI 工具，而是补充它们"。固定看板、OLAP 多维分析仍然走 Superset；对话 BI 解决的是"临时、模糊、高频"的查询需求。

**可能追问**：那 Superset + AI 插件不行吗？

### Q4. 4 类用户里哪个最重要？

**面试官意图**：考察优先级判断。

**参考答案**：

> **运营最重要**，三个原因：频次最高（日均 5-10 次）、决策时效性最强、也是研发最大的被打断源。

### Q5. "业务口径漂移每月约1次事故"具体什么情况？

**面试官意图**：验证真实性。

**参考答案**：

> 典型事故：月度报表里运营说"本月流失 200 人"，策划说"300 人"。原因是运营定义的"流失"是 `last_login_time ≥ 14 天`，策划定义的是 `last_action_time ≥ 7 天`。字段不同 + 天数不同 = 数字差 50%。管理层问"到底哪个对？"，两边各说各的理，最终研发被拉进来排查，浪费半天。
>
> 这直接推动了我设计"业务口径词典"——30+ 术语强制评审，"流失"统一为 `last_action_time ≥ 14 天 + 付费 ≥ P1`（和项目1 对齐）。词典变更走 Git + PM 评审，避免漂移。

**可能追问**：口径词典谁有权修改？

### Q6. 北极星指标为什么是 SSR 而不是查询准确率？

**面试官意图**：考察指标选择的商业逻辑。

**参考答案**：

> SSR（Self-Service Ratio，自助查询占比）直接对应"数据民主化"这个终极目标。准确率是基础门槛（达到 85% 就够用），不是增长指标——再往上投入 ROI 递减。
>
> SSR 的计算：自助完成的查询需求数 ÷ 所有数据查询需求数。基线 < 10%（几乎都走研发），MVP 目标 ≥ 50%，成熟期 ≥ 70%。
>
> 配套效率指标 QTT（Query Turnaround Time）：从问题提出到拿到可用结论的时长中位数。基线 1440 分钟（研发排期 1-2 天），MVP 目标 ≤ 5 分钟，实际 PoC 跑出 10 秒。

**可能追问**：SSR 怎么测量？"所有数据查询需求"这个分母怎么定？

## 2. 核心架构 · 两阶段 + 语义层

### Q7. 为什么选"两阶段+编译器"而不是直接 Text-to-SQL？这是全项目最重要的决策。

**面试官意图**：**核心题，验证架构决策能力。**

**参考答案**：

> Text-to-SQL 业界公认的准确率上限是 80%，在游戏 JSON 宽表场景下会降到 60% 以下——完全没法商用。
>
> 裸 Text-to-SQL 的失败点集中在：字段名拼错、JOIN 写错、JSON 路径错、忘记 LIMIT、忘记反引号、GROUP BY 漏列。
>
> 我的两阶段架构：
> - **Stage 1**：LLM 只输出结构化 Intent JSON（选择实体、度量、维度、过滤条件）
> - **Stage 2**：确定性编译器基于语义层 YAML 把 Intent 编译成 SQL
>
> 编译器天然不会拼错字段名（字段名来自枚举）、不会漏反引号（`payment-order-table` 的反引号已在 baseTable 里写好）、不会忘 LIMIT（强制补）、不会 JOIN 错表（JOIN 谓词预先写死）。
>
> **LLM 只负责"语义理解"这一件事**：把"上周 100 服流失的付费玩家"翻译成 `{entity:"player", measures:["churn_player_count"], filters:[{field:"server_id", op:"=", value:100}]}`。这比写完整 SQL 简单一个数量级。
>
> 代价是灵活性下降——语义层没覆盖的问题走不了 semantic path。所以我加了 **direct-SQL fallback** 兜底，保证产品不会卡死。

**可能追问**：你怎么说服团队接受这个方案？他们最初反对什么？

### Q9. 语义层 YAML 怎么设计的？5 个实体怎么选的？

**面试官意图**：考察语义层设计方法论。

**参考答案**：

> 从**高频查询需求倒推**。访谈收集 50+ 条真实查询，聚类发现 80% 围绕 5 个实体：
>
> | 实体 | 底表 | 维度数 | 度量数 | 覆盖查询类型 |
> |------|------|--------|--------|------------|
> | player | `players`（64列） | ~10 | ~6 | 玩家画像、流失、活跃、付费分层 |
> | payment | `payment-order-table` | ~3 | ~3 | 付费分析、ARPU/ARPPU |
> | guild | `guild` | ~2 | ~2 | 工会分析 |
> | guild_event | `guild_event` | ~2 | ~1 | 工会解散等事件 |
> | server | `server_list` | ~3 | — | 分服查询 |
>
> YAML 结构四层：Entity（对应底表）→ Dimensions（可 GROUP BY 的属性）→ Measures（可聚合的指标，带公式）→ Relations（跨实体 JOIN 路径）。
>
> 关键设计：**Virtual Columns**——把 `JSON_EXTRACT(vip, '$.level')` 封装成虚拟列 `vip_level`，LLM 只看到 `vip_level` 不需要知道底层是 JSON。这解决了"64 列 JSON 宽表"的核心问题。

**可能追问**：5 个实体覆盖不到的查询怎么办？

### Q10. 虚拟列解决什么问题？举个具体例子。

**面试官意图**：考察对 JSON 宽表问题的理解。

**参考答案**：

> `players.vip` 是一个 longtext JSON 字段，里面的结构是 `{"level": 7, "exp": 1234, ...}`。如果运营问"VIP 5 以上的玩家有多少？"，裸 Text-to-SQL 的 LLM 很可能写出 `WHERE vip >= 5`（把 JSON 当 int 用）或者 `WHERE JSON_EXTRACT(vip, 'level') >= 5`（漏了 `$` 前缀和 CAST）。
>
> 我在语义层里这样定义：
> ```yaml
> - name: vip_level
>   sql: CAST(JSON_EXTRACT(vip, '$.level') AS UNSIGNED)
>   type: int
>   comment: "虚拟列：从 vip longtext JSON 中提取 level"
> ```
>
> LLM 只需要在 Intent JSON 里写 `{field: "vip_level", op: ">=", value: 5}`，编译器自动展开成正确的 `CAST(JSON_EXTRACT(vip, '$.level') AS UNSIGNED) >= 5`。
>
> 类似的虚拟列还有 `current_guild_id`（从 `guild_info` JSON 提取）、`pay_tier`（CASE WHEN 表达式）、`register_date`（毫秒时间戳转日期）。这些是"LLM 最容易写错的 SQL 片段"，全部由编译器保证正确。

**可能追问**：JSON 结构如果变了，虚拟列定义要跟着改吗？

### Q11. relations 表的 JOIN 谓词为什么要预先写死？

**面试官意图**：考察细节设计的严谨性。

**参考答案**：

> 因为 Refantasia 的表设计有很多"反直觉"的地方，LLM 自己写 JOIN 大概率会错：
>
> 1. `payment-order-table` 的账号列叫 `account` 不是 `account_id`，而 `players` 的是 `account_id`。LLM 很容易写成 `players.account_id = payment.account_id`，多了个 `_id` 就查不出来。
> 2. `payment-order-table` 表名带连字符，必须用反引号 `` `payment-order-table` ``。
> 3. player → guild 的 JOIN 不是简单的外键，而是要从 `players.guild_info` JSON 里 `JSON_EXTRACT` 出 `guild_id` 再 JOIN。
>
> 这些细节在语义层的 relations 里预先写死：
> ```yaml
> - from: player
>   to: payment
>   join: "players.account_id = `payment-order-table`.account AND players.server_id = `payment-order-table`.server_id"
>   note: "account 列名不一样；表名带连字符必须反引号"
> ```
>
> 编译器遇到跨实体查询时，直接查 relations 表拼 JOIN，不让 LLM 自己写。

**可能追问**：如果需要三表 JOIN 呢？编译器支持吗？

### Q12. Intent 编译器怎么把 JSON 编译成 SQL？

**面试官意图**：考察编译器的实现思路。

**参考答案**：

> 编译器是一个确定性函数（不涉及 LLM），输入 Intent JSON，输出 SQL：
>
> ```python
> def compile(intent):
>     select = build_select(intent["measures"], intent["dimensions"])
>     from_  = resolve_joins(intent)  # 查 relations 表
>     where  = build_filters(intent["filters"])
>     group  = build_group(intent["dimensions"])
>     order  = build_order(intent.get("order_by"))
>     limit  = intent.get("limit", 1000)
>     return f"SELECT {select} FROM {from_} WHERE {where} GROUP BY {group} ORDER BY {order} LIMIT {limit}"
> ```
>
> 关键细节：
> - `measures` 的 aggregation 类型（count_distinct/sum/avg）在语义层 YAML 里定义，编译器自动展开。比如 `churn_player_count` 带 `filter: "last_action_time ≥ 14 天"`，编译器会生成 `COUNT(DISTINCT CASE WHEN ... THEN account_id END)`。
> - `filters` 类型化 quoting 防注入：int 不加引号，string 加引号并转义。
> - LIMIT 强制 ≤ 1000，如果 Intent 没写就默认补 1000。
>
> 这就是"SQL 正确率 100%"的保证——编译器是确定性代码，不存在"有时候对有时候错"。

**可能追问**：编译器有 bug 怎么办？你怎么测试它？

### Q13. semantic path 和 direct-SQL fallback 怎么选择？

**面试官意图**：考察双路径设计。

**参考答案**：

> Stage 1 的 LLM 输出里有一个 `mode` 字段：
> - `mode=semantic`：意图可以映射到语义层的实体/度量/维度 → 走编译器。
> - `mode=direct`：超出语义层覆盖范围（比如"给我看 conquest_history 表的前 10 行"）→ LLM 直接生成 SQL，走传统 Text-to-SQL 路径。
>
> 选择逻辑：
> 1. LLM 在 Stage 1 Prompt 里被告知"优先用 AVAILABLE_ENTITIES 里的实体/度量，如果覆盖不了则返回 `mode=direct`"。
> 2. 即使 LLM 返回了 `mode=semantic`，如果编译器报错（比如引用了不存在的度量），也自动降级到 direct 路径。
> 3. direct 路径仍然经过 9 层 SQL 安全沙盒——不安全的 SQL 照样被拦。
>
> 预期流量分配：语义层覆盖 70-80% 的查询（MVP 阶段），direct fallback 覆盖剩余 20-30%。随着语义层扩展，semantic path 占比会逐步提升。

**可能追问**：direct fallback 的准确率只有 80%，用户能接受吗？

### Q14. 业务口径词典 30+ 术语怎么管理？

**面试官意图**：考察知识管理方法论。

**参考答案**：

> 三个原则：
>
> 1. **三方评审**：每条术语首次进入词典需 PM + 数据工程师 + 业务方三方确认。比如"流失"这个词，运营、策划、项目1 的定义必须对齐成一个。
> 2. **Git 管理**：词典是 YAML 文件，变更走 Git PR。每次修改有 changelog、reviewer、原因说明。
> 3. **自动校验**：语义层回归测试在每次词典变更时自动触发——确保所有 measure 的 SQL 还能编译和执行。
>
> 核心术语举例：
> - "流失" = `last_action_time ≥ 14 天 + 付费 ≥ P1`
> - "工会解散" = `guild_event.event_type = 9`（int 枚举，不是字符串 'DISBAND'）
> - "大 R / 中 R / 小 R" = P0 / P1 / P2，与项目1 对齐
> - "ARPU" = `SUM(pay_money) / COUNT(DISTINCT account_id)`
>
> 词典的价值不只是对话 BI，项目1/2 也消费同一份——这是"AI 产品矩阵"的数据标准层。

**可能追问**：如果业务方不同意统一口径怎么办？

### Q15. few-shot 样例怎么选的？

**面试官意图**：考察 Prompt 工程细节。

**参考答案**：

> Stage 1 Prompt 注入 3-5 条 (question, intent_json) 样例。选择逻辑：
>
> 1. **按查询复杂度分级**：至少包含一条简单聚合（"100 服玩家数"）、一条带过滤的（"上周流失的付费玩家"）、一条跨实体 JOIN（"工会解散导致流失的玩家"）。
> 2. **覆盖常见歧义**："流失"在 `resolved_terms` 里对齐到口径词典 key，让 LLM 学会做术语消歧。
> 3. **动态注入**：不是固定 5 条，而是根据用户问题做 embedding 匹配，从"样例库"（~50 条标注好的 question-intent 对）里取最相似的 3-5 条。这样每次 Prompt 的 few-shot 都和当前问题最相关。
>
> 样例库的维护：每周从 👎 反馈中挑出"修正后正确的"query 加入样例库。样例库越来越大 → few-shot 命中率越来越高 → 准确率持续提升。这是反馈闭环的核心。

**可能追问**：动态选 few-shot 的 embedding 模型用的什么？

### Q16. 语义层编译器保证 SQL 正确率 100% 怎么做到的？

**面试官意图**：追问"100%"这个强承诺。

**参考答案**：

> "100%"指的是**语法正确 + 字段存在 + JOIN 正确 + LIMIT 存在**——这些是编译器可以保证的。不保证**语义正确**（用户问的和 LLM 理解的可能不一样）。
>
> 编译器保证的四件事：
> 1. **字段名 100% 正确**：所有字段来自语义层 YAML 枚举，不存在"拼错字段名"。
> 2. **JOIN 100% 正确**：跨实体查询的 JOIN 谓词来自 relations 预定义，不存在"ON 条件写错"。
> 3. **JSON 路径 100% 正确**：虚拟列的 `JSON_EXTRACT` 路径在 YAML 里写死，不存在"漏 $"。
> 4. **LIMIT 一定存在**：编译器强制补 LIMIT 1000。
>
> 但如果 LLM 在 Stage 1 把"流失"理解成了"活跃"（intent 错误），编译器生成的 SQL 虽然语法正确但结果是错的——这就是"意图解析正确率 ≥ 90%"这个指标要衡量的。编译器管"SQL 正确性"，LLM 管"意图正确性"，各司其职。

**可能追问**：如果语义层 YAML 本身定义错了呢？

### Q17. 两阶段的代价是灵活性下降，你怎么评估？

**面试官意图**：考察权衡取舍的思维。

**参考答案**：

> 灵活性下降的表现：语义层没覆盖的实体/度量/维度，走不了 semantic path。比如用户问"给我看 conquest_history 原始表"，语义层没有 conquest_history 这个实体，只能走 direct-SQL fallback。
>
> 我的评估方法：用 50 条真实查询需求做覆盖率分析——其中 38 条（76%）可以完全映射到 5 个实体的语义层定义，12 条需要 direct fallback。
>
> **76% 的覆盖率够不够？** 我认为 MVP 够了，原因：
> 1. 这 76% 是高频查询——运营和策划日常问的大多是"流失/付费/活跃/工会"这四大主题。
> 2. 剩下 24% 走 direct fallback，虽然准确率只有 80%，但加上 9 层安全沙盒，不会出恶性问题。
> 3. 语义层是可以持续扩展的——每周 PM 评审新增 1-2 个实体/度量，覆盖率会逐步提升到 90%+。
>
> 核心权衡：**用 76% 查询的 100% 正确率 + 24% 查询的 80% 正确率，换来了整体产品的可信度**。如果全走裸 Text-to-SQL，100% 查询都是 60-80% 正确率，用户第一次碰到错误就不信任了。

**可能追问**：语义层从 76% 扩展到 90% 要多久？

### Q18. Stage 1 的 Prompt 怎么设计的？

**面试官意图**：考察 Prompt 工程。

**参考答案**：

> Stage 1 Prompt 的核心结构：
>
> ```
> SYSTEM: 你是 Refantasia 游戏数据助手。把用户问题翻译为结构化查询意图 JSON。
>         禁止自己写 SQL。只能使用 AVAILABLE_ENTITIES 里的实体/维度/度量。
>
> AVAILABLE_ENTITIES: {语义层摘要 YAML}
>
> BUSINESS_GLOSSARY: {口径词典子集}
>
> OUTPUT_SCHEMA: {Intent JSON Schema}
>
> FEW-SHOT: 3-5 条 (question, intent) 样例
>
> USER: {用户问题}
> ```
>
> 关键设计：
> 1. **"禁止自己写 SQL"**：硬约束，防止 LLM 绕过语义层。
> 2. **AVAILABLE_ENTITIES 是摘要不是全量**：只注入和问题相关的实体定义，控制 token 量。
> 3. **BUSINESS_GLOSSARY 注入**：让 LLM 碰到"流失"时自动消歧。
> 4. **OUTPUT_SCHEMA 里有 `resolved_terms` 字段**：要求 LLM 标明"我把哪个词消歧成了哪个口径定义"，方便审计。

**可能追问**：如果用户的问题完全超出语义层范围，LLM 会返回什么？

## 3. SQL 安全与权限

### Q19. SQL 9 层安全沙盒每层做什么？

**面试官意图**：考察安全设计的系统性。

**参考答案**：

> ```
> Layer 1  语法解析    sqlparse 解析，确保单语句
> Layer 2  白名单      只允许 SELECT/SHOW/DESCRIBE/EXPLAIN
> Layer 3  禁词        INSERT/UPDATE/DELETE/DROP/TRUNCATE/ALTER/CREATE/GRANT
> Layer 4  强制 LIMIT  缺失则自动补 LIMIT 1000
> Layer 5  EXPLAIN     预估 rows，> 1,000,000 拒绝
> Layer 6  PII 脱敏    name/email/phone/device_id → SHA2()
> Layer 7  只读副本    专用只读账号，表级权限白名单
> Layer 8  超时        执行超时 30 秒自动 kill
> Layer 9  并发限制    单用户最多 3 条活跃查询
> ```
>
> MVP 阶段落地了前 4 层（语法/白名单/禁词/LIMIT），V2 规划后 5 层。前 4 层已经能拦住 95% 的安全风险——因为 semantic path 下编译器生成的 SQL 天然安全，只有 direct fallback 路径才可能有问题。

**可能追问**：9 层会不会太多？有没有过度设计？

### Q20. PII 脱敏是在 SQL 层做还是结果层做？

**面试官意图**：考察脱敏策略。

**参考答案**：

> **SQL 层做**——在 SQL 执行前把 PII 字段替换成 `SHA2()` 函数：
>
> ```sql
> -- 原始 SQL
> SELECT name, server_id, vip_level FROM players WHERE ...
> -- 脱敏后
> SELECT SHA2(name, 256) AS name, server_id, vip_level FROM players WHERE ...
> ```
>
> 为什么不在结果层做？因为结果层脱敏意味着 PII 明文已经从 DB 传到了应用层——即使最终展示时打星号，中间链路有泄露风险。SQL 层脱敏意味着 PII 明文**从未离开 DB**。
>
> 例外：客服角色（support profile）查单玩家明细时不脱敏——因为客服的 SLA 要求看到真实玩家名来处理工单。这是 PRD 里评审通过的决策。

**可能追问**：SHA2 不可逆，如果运营需要根据脱敏结果反查玩家怎么办？

### Q21. EXPLAIN 预估 rows > 100 万就拒绝，为什么是 100 万？

**面试官意图**：考察阈值设定。

**参考答案**：

> 三个依据：
>
> 1. **当前数据规模**：`players` 表 2491 行，最大的 `configurations` 12608 行。正常查询扫描行数在几千级别。100 万是当前最大表的 80 倍，如果 EXPLAIN 估出 100 万，大概率是全表笛卡尔积——说明 SQL 写错了。
>
> 2. **DB 性能边界**：只读副本是小规格实例，100 万行 SELECT 可能要 10-30 秒，接近 30 秒超时阈值。拒绝比执行后超时更友好——用户能看到"查询太大，请缩小范围"的提示。
>
> 3. **渐进式放开**：100 万是 MVP 的保守值。上线后根据真实查询的 EXPLAIN 分布调整——如果发现 90% 的正常查询都在 10 万以下，可以收紧到 50 万。

**可能追问**：EXPLAIN 的估算准确吗？MySQL 的 EXPLAIN rows 有时候偏差很大。

### Q22. 4 级角色权限怎么设计的？

**面试官意图**：考察权限模型。

**参考答案**：

> | 角色 | 数据范围 | PII 脱敏 | 特殊限制 |
> |------|---------|---------|---------|
> | 运营（analyst） | 全部业务表，聚合结果 | ✅ 强制脱敏 | 禁止 `SELECT *` / 单玩家 WHERE / 无聚合函数 |
> | 策划（analyst） | 同上 | ✅ 强制脱敏 | 同上 |
> | 客服（support） | 单玩家明细 | ❌ 不脱敏 | 只允许带 `WHERE account_id = ?` 的单行查询 |
> | 管理层（analyst） | 聚合结果 | ✅ 脱敏 | 同运营 |
>
> 两层联动：SQL 没执行前就能拦掉越权请求。比如运营的 SQL 里出现 `WHERE account_id = 'xxx'`（单玩家查询），在 Layer 5 就被拦住——"运营角色不允许查看单玩家明细，请使用聚合查询"。

## 4. 结果解读与 Agent 深挖

### Q23. Agent 深挖建议是怎么生成的？

**面试官意图**：考察 Agent 设计。

**参考答案**：

> 基于"结果模式"匹配策略表 + LLM 改写：
>
> | 结果模式 | 深挖建议模板 |
> |---------|------------|
> | 计数单值 | "按服务器拆分"、"按付费等级拆分"、"与上周对比" |
> | 时间序列 | "找出峰值日"、"是否与某活动重合"、"预测下周" |
> | TOP N | "看 TOP 用户画像"、"看 BOTTOM 用户差异" |
> | 对比类 | "归因：哪个子维度贡献最大" |
>
> 实现流程：
> 1. 规则匹配产出 3-5 条候选建议（模板填充）。
> 2. LLM 过滤 + 改写为自然语言问题（"上周 100 服流失的付费玩家按工会拆分是什么分布？"）。
> 3. 每条建议**预编译好 Intent JSON**——用户点击时跳过 Stage 1 LLM，直接走编译器，秒出结果。
>
> 点击率目标 ≥ 25%。这个数字参考了 Google 搜索"People also ask"的点击率（约 3-8%），但对话 BI 的场景更聚焦，25% 是合理预期。

**可能追问**：如果用户从来不点深挖建议怎么办？

### Q24. 查询结果为 0 行你怎么处理？

**面试官意图**：考察边界情况处理。

**参考答案**：

> 0 行结果不等于"系统错误"，但用户会困惑"为什么没数据"。我的处理：
>
> 1. **LLM 结果解读主动说明可能原因**：Prompt 里有硬约束"如果数据量为 0，提示可能的原因"——比如"可能是时间范围不对（您查的是上周但 100 服上周没有活动）"或"口径条件太严格（付费 ≥ P1 且流失 ≥ 14 天同时满足的玩家本来就少）"。
>
> 2. **Agent 自动建议放宽条件**：深挖建议变成"放宽时间范围到本月"、"去掉付费等级过滤"、"换成全服而不是单服"。
>
> 3. **展示 SQL + 口径说明**：让用户看到"系统理解的是什么"——如果口径理解错了，用户可以手动修正。

**可能追问**：如果是 SQL 本身写错了导致 0 行呢？

### Q27. 三层评测怎么做？

**面试官意图**：考察评测体系。

**参考答案**：

> ```
> Layer 1: CI 自动评测（每次 Prompt/语义层提交触发）
>   ├─ Intent JSON diff（结构等价判断）
>   ├─ SQL AST diff（sqlparse 归一化后对比）

>   目标：👍 率 ≥ 85%
> ```

### Q28. 影子模式是什么？为什么需要？

**面试官意图**：考察上线安全机制。

**参考答案**：

> 新版本 Prompt 上线前，先开启影子模式：
> - 线上请求同时走新旧两个 Prompt（并行）
> - 只把**旧版**结果返给用户（用户无感知）
> - 新版结果异步落库对比
> - 连续 3 天一致率 ≥ 98% → 切流到新版
>
> 为什么需要？因为 Prompt 改一个词可能导致 10% 的查询行为变化——回归测试集只有 200 条，覆盖不了所有真实场景。影子模式用"线上真实流量"做验证，比回归测试更可靠。
>
> 代价是 LLM 成本翻倍（每个请求调两次），但只持续 3 天，可以接受。

**可能追问**：如果 3 天一致率只有 95% 怎么办？

### Q29. 反馈闭环"👎 在 7 日内闭环"你怎么保证？

> ```

### Q35. 如果让你重做，你会改什么？

### Q36. 从这个项目你学到的最重要的一课？

**面试官意图**：收尾题。

**参考答案**：

> **"让 LLM 做得更少，而不是更多。"**
>
> 做这个项目之前，我的直觉是"LLM 越强越好、给它越多能力越好"。做完之后我的认知反转了：LLM 的价值在于"理解人话"这一件事，其余的（生成 SQL、保证安全、字段正确性）都应该交给确定性代码。
>
> 两阶段架构的本质就是"限制 LLM 的输出空间"：从"任意 SQL 字符串"收窄到"有限 schema 的 Intent JSON"。输出空间越小，幻觉越少，可验证性越强。
>
> 这个认知也影响了项目1（归因用规则层打底，LLM 只做复核）和项目2（术语用词典强制替换，LLM 只做识别）。三个项目的共同哲学是：**给 LLM 一个尽可能小的"活动范围"，然后在这个范围内让它发挥最大价值。**
